# Phase 12 — UI Design Contract

> Visual and interaction contract for Phase 12 (Section Parity & Polish). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Platform:** native macOS (AppKit/SwiftUI). No web, no Tailwind, no shadcn. Design tokens are SF Symbols, SwiftUI `.font(.system(size:weight:))`, `.font(.<role>)`, and `Theme.*` (NSColor) — not hex / CSS variables. The template shape is preserved; values are substituted for the native stack.
>
> **Phase posture:** normalization + audit pass. Every visual decision already locked by prior phases; this phase's job is to document the canonical shapes so the Planner and Executor implement without drift. **No new visual language is introduced.**

## Spacing Scale

Phase 12 introduces no new spacing tokens. All values below are multiples of 2; multiples of 4 are the default step, with 2 reserved for tight sidebar list-row insets where density matters (per `.sidebarRowSize(.small)`).

| Token | Value | Usage |
|-------|-------|-------|
| xxs | 2px | Vertical padding inside `listRowInsets` tight rows (existing sidebar rhythm) |
| xs | 4px | Icon-to-label gaps; horizontal `listRowInsets`; chevron-to-title spacing in `SidebarSectionHeader` |
| sm | 8px | Compact element spacing; overview header chevron-to-title spacing (`CollapsibleVStackSection` `HStack(spacing: 8)`) |
| md | 12px | Overview card internal padding (`CardStyle` — existing); `CollapsibleVStackSection` outer `VStack(spacing: 12)` between header and content |
| lg | 16px | Default element spacing; alert message body vertical rhythm (system default) |
| lgx | 24px | `ProjectOverviewView` top-level `VStack` spacing between sections (existing `spacing: 24`) |
| xl | 32px | Overview scroll-view padding (existing `.padding(32)`) |

Exceptions:
- `.sidebarRowSize(.small)` + `.defaultMinListRowHeight(20)` are already applied at `SidebarView.swift:100-101`. All canonical `SidebarSectionHeader` call sites (Scratch Pads, Locations, Browsers, Other Sessions, nested Bookmarks) inherit these — **no per-section row-height override** (locks UIX-20 pixel-identical row height).
- **Count badge** sits immediately after the title inside the header's existing `HStack(spacing: 4)` — no additional padding between title text and count text; the 4pt `HStack` spacing is the gap. This matches Phase 11's shipped bespoke sub-header verbatim (`NestedBookmarkSubSection.swift:44-56`).
- **Count right-alignment:** count does NOT right-align. It renders left-adjacent to the title, before the trailing `Spacer()`. Prevents the count from drifting with variable sidebar widths and matches the shipped pattern.
- **Alert internal spacing:** system default — SwiftUI `.alert()` manages its own layout; no custom padding overrides. Button order follows SwiftUI defaults (destructive trailing, Cancel leading). This is macOS convention.
- **Inline rename TextField:** reuses existing `.textFieldStyle(.roundedBorder)` at `ScratchPadSection.swift:63`; no padding adjustments. The `.roundedBorder` style supplies the standard macOS field inset.

Two new rule blocks land in `CLAUDE.md` this phase. Both document invariants the Planner/Executor must encode; neither drives runtime behavior.

**Rule 1 — `@AppStorage` naming convention (D-15, UIX-22):**

> **Sidebar/Overview section expansion keys (UIX-22).** Any `@AppStorage` key covering sidebar or project-overview section expansion state MUST be declared in `SidebarStorageKeys` or `OverviewStorageKeys` — never inline as a string literal. Sidebar keys follow `sidebar.<section>.expanded` (global, not per-project). Overview keys follow `overview.<section>.expanded.<projectID>` (per-project, built via the constants-file helpers). ## Interaction Contracts (phase-specific)

Four interaction patterns land this phase. All consume existing code surfaces — no new SwiftUI primitives introduced.

### Pattern A — Canonical `SidebarSectionHeader` with count (UIX-20)

**Component contract (post-edit):**

```swift
struct SidebarSectionHeader: View {
    let title: String
    @Binding var isExpanded: Bool
    let count: Int?   // NEW — default nil

    var body: some View {
        HStack(spacing: 4) {
            Button {
                withAnimation(.snappy(duration: 0.15)) {
                    isExpanded.toggle()
                }
            } label: {
                HStack(spacing: 4) {
                    Image(systemName: "chevron.right")
                        .font(.system(size: 9, weight: .semibold))
                        .rotationEffect(.degrees(isExpanded ? 90 : 0))
                    Text(title)
                    // Count rendered only when collapsed AND count is non-nil
                    if let count, !isExpanded {
                        Text("\(count)")
                            .font(.system(size: 10))
                            .foregroundStyle(.tertiary)
                    }
                }
                .contentShape(Rectangle())
            }
            .buttonStyle(.plain)
            Spacer()
        }
    }
}
```

**Invariants:**
- Chevron size/weight unchanged (9pt semibold).
- Chevron rotation animation unchanged (`snappy(duration: 0.15)`).
- Count **only** renders when `count != nil && !isExpanded`. Expanded headers show only the title.
- Count uses `.font(.system(size: 10))` + `.foregroundStyle(.tertiary)` — **no other scale acceptable**.
- No trailing accessory, no action button — the header is navigation-only per Phase 10 UIX-03.
- `count: Int?` defaulting to `nil` is the preferred ergonomic (D-168 discretion rejected in favor of nil-Optional for explicit intent).

**Call-site parity (five sections):**

| Call site | File:line | `count:` value |
|-----------|-----------|---------------|
| Scratch Pads | `SidebarView.swift:~49-60` | `visiblePads.count` (lifted from `ScratchPadSection`; see Integration Points) |
| Locations | `SidebarView.swift:~72-92` | `projectLocations.count` |
| Browsers | `SidebarView.swift:~534-565` (`BrowserSessionsSection`) | `browsers.count` — **tabs only, NOT including bookmarks** (D-02) |
| Other Sessions | `SidebarView.swift:~569-587` (`OrphanedSessionsSection`) | `orphaned.count` |
| Nested Bookmarks sub-section | `NestedBookmarkSubSection.swift:~39-59` (replaces bespoke header) | `visibleBookmarks.count` |

**Retired element:** the bespoke inline `Button { … } label: { HStack { … } }` header in `NestedBookmarkSubSection.swift:39-59` is deleted. Its `.onDrop(of: [.text], delegate: BookmarkFolderDropDelegate(targetFolderID: nil, modelContext: modelContext))` modifier **must reattach** to the new `SidebarSectionHeader(title: "Bookmarks", isExpanded: $isExpanded, count: visibleBookmarks.count)` call site. ### Pattern C — Selection-driven rename/delete (UIX-21, Scratch Pad ↔ Bookmark parity)

**Selection model** (D-06): SwiftUI `List(selection:)` binding scoped to `SidebarView`. Rows in the two participating sections (Scratch Pads + nested Bookmarks) declare `.tag(…)` to participate. Other sections (Browser rows, Locations, Other Sessions, Overview row) do not gain selection — blast-radius stays small.

**Selection visual:** inherits default macOS `List(selection:)` background treatment, composed with the existing `Theme.sidebarSelectionBackground(isSelected:)` where rows already specify one (e.g., `ScratchPadSection.padRow:43` uses `isActive`). Rows must OR the two states so an active + selected pad paints once, not twice.

**Keyboard contract (identical for both row types):**

| Key | Action | Precondition |
|-----|--------|--------------|
| Return | Enter rename flow | Exactly one row selected |
| Delete | Show `.alert()` confirmation → on confirm, delete | Exactly one row selected |
| Esc (during rename) | Cancel rename, restore original name | Row is currently in rename mode |
| Return (during rename) | Commit rename, clear selection | Row is currently in rename mode, `renameText` is non-empty after `trimmingCharacters(in: .whitespaces)` |

**Row-specific rename behavior (D-07):**

| Row type | Rename surface |
|----------|---------------|
| Scratch Pad | **Inline `TextField`** — reuses the existing flow at `ScratchPadSection.swift:17-23, 59-66`. `renamingURL = pad.url` on Return keypress; `.textFieldStyle(.roundedBorder)`; `onCommit` → `commitRename`; `onExitCommand` → cancel. Only the **trigger** changes (selection + Return, not context-menu click). |
| Bookmark | **`BookmarkEditSheet`** — reuses the existing sheet. Return on a selected bookmark row sets `editing = bookmark`, which presents the sheet via the existing `.sheet(item: $editing)` modifier attached to the `NestedBookmarkSubSection` scope (`:70-72`). |

**Rename TextField focus treatment (Scratch Pad only):** reuses shipped `.textFieldStyle(.roundedBorder)` — **do NOT adjust**. The rounded border + system focus ring is the canonical macOS treatment and already passes Phase 11's visual audit. Select-all on entry follows Finder convention (Claude's discretion, D-168) — planner implements via `@FocusState` + `NSTextView` selectAll or equivalent SwiftUI focus.

**Row-specific delete behavior (D-08, D-10):**

| Row type | Keyboard Delete | Context-menu Delete | Underlying operation |
|----------|----------------|---------------------|---------------------|
| Scratch Pad | `.alert()` confirm with `Move '\(pad.name)' to Trash?` → on "Move to Trash", `NSWorkspace.shared.recycle([url])` | **Immediate, no confirmation** (D-09) — `NSWorkspace.shared.recycle([url])` | Soft-delete to macOS Trash; recoverable from Finder (D-10). Matches macOS convention. |
| Bookmark | `.alert()` confirm with `Delete bookmark '\(displayName)'?` → on "Delete", `BookmarkStore.remove(bookmark)` | **Immediate, no confirmation** (D-09) — `BookmarkStore.remove(bookmark)` | SwiftData hard-delete via `modelContext.delete`. |

**Confirmation rationale (D-09):** context-menu click is a deliberate motion (no friction warranted); keyboard Delete is easy to mispress (confirm). The hybrid applies identically to both row types.

**Retired element:** `DeleteKeyHandler` private struct at `NestedBookmarkSubSection.swift:336-349` is **deleted entirely** (D-11). The hover-based `.focusable(enabled).onKeyPress(.delete)` pattern is gone; the selection-driven flow replaces it.

**Selection-clear invariant (D-12):**
- After a rename commits (Scratch Pad `onCommit` or Bookmark sheet dismiss), selection must clear. Prevents subsequent Return from re-entering rename mode on the same row.
- After a delete confirms, selection must clear (row no longer exists — clearing prevents a stale selection highlighting the next row).

**Fallback (code_context risk):** if `List(selection:)` across heterogeneous Sections proves visually flaky on macOS (unexpected highlights, cross-section selection leak), the Planner falls back to per-section `@State` scoped to `ScratchPadSection` and `NestedBookmarkSubSection`. **The keyboard contract above is identical either way** — only the plumbing changes. Deviation must be noted in the Plan SUMMARY.

## Accessibility & Keyboard

Phase 12 explicitly **expands** keyboard affordances — parity with existing Bookmarks hover-delete was the prior state; selection-driven rename/delete is the new canonical state.

**New/changed keyboard flows:**
- **Selected Scratch Pad + Return** → enter inline rename mode. `TextField` gains keyboard focus; subsequent Return commits, Esc cancels.
- **Selected Scratch Pad + Delete** → `.alert()` confirmation → `NSWorkspace.recycle` on confirm.
- **Selected Bookmark + Return** → opens `BookmarkEditSheet` (full rename + URL + folder edit).
- **Selected Bookmark + Delete** → `.alert()` confirmation → `BookmarkStore.remove` on confirm.

**Preserved:**
- All existing context-menu affordances on both row types (Rename, Reveal in Finder, Open in New Tab, Copy URL, Edit..., Delete, etc.).
- File menu shortcuts (`Cmd+Option+B`, etc.) unchanged — Phase 10 scope.
- Focus-scoped browser shortcuts unchanged — Phase 2-3 scope.
- Chevron toggle keyboard-focusable (inside `Button { … } .buttonStyle(.plain)` — unchanged).

**VoiceOver:**
- Canonical `SidebarSectionHeader` reads: "`<title>`, `<count>` items, collapsed, button" when collapsed with count; "`<title>`, expanded, button" when expanded. SwiftUI composes this from the button label's `HStack` children.
- Count text is not individually focusable — it's inside the header button's label, so VoiceOver traverses it as part of the combined element.
- Alert dialogs use system NSAlert surface — full VoiceOver support is built-in.
- Inline rename TextField announces as "Name, text field, editing" (SwiftUI default).

**Invariants:**
- Selection highlight is not the only selection cue for accessibility — SwiftUI `List(selection:)` provides accessibility state automatically.
- Destructive buttons (`role: .destructive`) are announced as "destructive" by VoiceOver — no custom accessibility label needed.
